# 教案功能需求文档

## 1. 文档定位

这份文档只定义教案功能“应该做什么”，不定义实现细节。

如果以后教案链发生以下变化，必须先改本文件：

- 用户可见行为变化
- 输入字段语义变化
- 输出结果语义变化
- 教案结构定义变化
- 验收标准变化

如果只是实现方式改变，不需要先改本文件。

## 2. 功能目标

教案链的目标是：根据课程、课时、附加要求、字数限制、教材或知识库内容，生成一份结构完整、逻辑清晰、可继续向下游复用的教学教案。

这条链的最终价值不只是“输出一篇文章”，而是要输出一个可复用的教学设计产物。

教案链至少应满足：

1. 能独立完成教案生成
2. 能为试卷和 PPT 提供稳定的上游上下文
3. 能尽量利用教材、知识库和上传资料
4. 能把结构化约束落实到输出中

## 3. 用户可见输入

根据旧系统现状，教案链至少应支持以下业务语义：

- `course`
  课程名称
- `units`
  单元名称
- `lessons`
  课时名称
- `constraint`
  附加要求
- `word_limit`
  字数限制
- `use_rag`
  是否使用知识库检索增强
- `model_type`
  模型类型
- `enable_thinking`
  是否输出思考过程

这些字段来自旧代码：

- `/Users/sss/directionai/DirectionAICloud/evoagentx/evo_modules/lesson_generator.py`

重构后，字段命名可以在兼容层内部映射，但业务语义必须保持稳定。

## 4. 用户可见输出

### 4.1 最终结果

用户至少应能看到：

- 一份完整教案正文
- 教案生成过程的思考步骤
- 生成完成时间
- 如有需要，可导出或传递给下游 PPT / 试卷

### 4.2 当前前端已依赖的完成态字段

当前旧前端对 lesson SSE 完成态至少依赖：

- `lesson_plan`

这意味着：

- 新后端内部可以完全更换 artifact 结构
- 但兼容层最终必须仍然能向前端提供 `lesson_plan`

## 5. 当前前端已依赖的流式行为

当前前端 store：

- `/Users/sss/directionai/DirectionAICloud/nginx/src/stores/modules/lesson-generation.ts`

已依赖的 SSE 事件语义包括：

- `thinking_start`
- `thinking_chunk`
- `thinking_end`
- `progress`
- `done`
- `error`

因此，教案链重构后必须继续支持这些事件语义。

其中：

- `progress` 用于驱动思考步骤和调度提示
- `done` 至少要给出 `lesson_plan`

## 6. 教案内容要求

教案不是任意文章，而应尽量覆盖教学设计的核心部分。

推荐至少包含：

- 课程主题
- 教学目标
- 教学重点
- 教学难点
- 教学方法
- 教学准备
- 教学过程
- 理论内容
- 实践或互动设计
- 小结
- 作业或延伸任务

如果课程类型适合，还应支持：

- 实验环节
- 项目制教学设计
- 案例分析环节
- 双语内容或专业术语解释

## 7. 教案结构要求

### 7.1 结构要稳定

教案链输出不应每次结构完全漂移。

即使最终前端展示的是 markdown，内部也必须产出结构化 artifact，避免后续只能靠正则去抽取。

### 7.2 结构要可下游复用

教案链输出必须适合作为下游输入，至少要能让下游方便拿到：

- 教学主题
- 目标受众
- 章节结构
- 关键知识点
- 教学流程摘要
- 活动设计摘要
- 核心概念说明

## 8. 知识增强要求

### 8.1 RAG 是可选增强，但不是唯一依赖

如果开启 `use_rag`：

- 应优先利用知识库增强输出质量
- 不应让检索失败直接导致整条链不可用

如果关闭 `use_rag`：

- 教案链也应能正常生成

### 8.2 上传文档是有效上游来源

如果用户提供教材、讲义或其他文档：

- 文档内容应能参与教案生成
- 教案应尽量体现上传内容的重点

## 9. 与其他链路的关系

### 9.1 教案到试卷

教案可以作为试卷蓝图的重要输入来源之一，尤其适合提供：

- 知识点覆盖范围
- 难点分布
- 重点内容

### 9.2 教案到 PPT

教案可以作为 PPT 的上游上下文，尤其适合提供：

- 章节层次
- 每部分核心内容
- 教学流程

### 9.3 不要求反向依赖

默认不要求：

- 试卷先生成再反向约束教案
- PPT 先生成再反向约束教案

教案在组合任务中通常是上游，不是下游。

## 10. 失败与降级要求

### 10.1 不能静默失败

如果教案生成失败：

- 前端应能看到失败
- 不应只表现成“没结果”

### 10.2 应允许局部回炉

如果问题只出在某一部分：

- 应优先局部修复
- 不要求整条链完全重跑

### 10.3 RAG 不可用时允许降级

如果知识库或检索工具不可用：

- 应优先降级生成
- 不要把整个教案链直接打死

## 11. 非功能性要求

### 11.1 可维护性

教案链必须摆脱旧硬编码 workflow 的强绑定，做到：

- 结构清晰
- schema 清晰
- 模块边界清晰

### 11.2 可调试性

应能知道：

- 当前执行到哪个阶段
- 哪一步失败
- 是否启用了 RAG
- 产出了哪些 artifact

### 11.3 可测试性

应能对以下内容做测试：

- request schema
- SSE contract
- artifact 结构
- 有 / 无 RAG 场景
- 复杂约束场景

## 12. 验收标准

教案链重构完成的最低标准：

1. 旧前端可以继续消费 lesson 生成结果
2. `done` 事件仍能提供 `lesson_plan`
3. 内部不再依赖旧硬编码 workflow 才能工作
4. 内部存在结构化 lesson artifact
5. 可以作为试卷和 PPT 的上游上下文
6. 有基本 contract test、integration test 和 regression test

## 13. 非目标

本阶段不要求：

- 完成所有教研场景的最优质量
- 支持无限多教学模板
- 一次性重做所有 lesson 评估系统

本阶段重点是：

- 先把教案链重构成清晰、稳定、可持续迭代的架构
