# AI 后端需求文档

## 1. 文档定位

这份文档定义的是“新 AI 后端应该做到什么”，而不是“当前代码是怎么写的”。

当前仓库就是正式重构仓库：

`directionai-agent-backend`

只要涉及用户可见行为、对前端的兼容要求、三类核心能力的最终目标，都应以这份文档为准。

如果以后修 bug 时发现：

- 团队对“正确行为”理解不一致
- 某个返回结果到底该不该变不清楚
- 某个流程是否允许自动执行说不清

应先修改这份文档，再修改代码。

## 2. 项目目标

要建设一个可长期维护的 AI 教育后端，支持以下三类核心生成能力：

1. 教案生成
2. 试卷生成
3. PPT 生成

这个后端未来的主要职责不是“多调用几个模型接口”，而是形成稳定的教育内容生产系统，具备以下特征：

- 输入来自结构化表单
- 输出可直接被前端展示和下载
- 内部可重构为 Agent、Tool、Skill、Artifact、Trace 驱动的架构
- 对外行为稳定，不因为后端重写而迫使前端跟着改
- 对复杂生成任务具备一定的自纠错、评测、回炉和可追踪能力

## 3. 重构边界

### 允许重构的部分

- AI 后端内部流程
- 模型调用方式
- Prompt 组织方式
- 教案和试卷的生成链路
- RAG 接入方式
- 工具封装方式
- 状态管理和日志追踪方式
- 后端仓库组织结构

### 默认不允许随意变化的部分

- 前端页面结构
- 前端表单输入方式
- 前端结果展示方式
- 前端已依赖的流式输出行为
- 前端已依赖的下载、预览、状态显示和最终结果呈现

## 4. 对前端的稳定约束

### 4.0 当前前端已用入口

根据旧仓 `/Users/sss/directionai/DirectionAICloud` 中的前端代码，当前至少已经有以下关键入口在被消费：

- `nginx/src/api/modules/bitot.ts`
- `nginx/src/stores/modules/lesson-generation.ts`
- `nginx/src/stores/modules/exam-generation.ts`
- `nginx/src/stores/modules/ppt-generation.ts`

从这些代码看，当前前端直接依赖的关键流式入口包括：

- `/evoapi/stream_lesson_plan`
- `/evoapi/stream_exam`
- `/pptagentapi/stream_ppt`
- `/pptagentapi/stream_evaluate/ppt`
- `/evoapi/upload_document`

因此，新后端至少要通过 Compatibility API 保证这些入口背后的业务语义不变。

### 4.1 页面与交互保持稳定

重构后，前端不应因为 AI 后端重构而改页面或重新设计交互流程。

这意味着：

- 用户仍通过原有表单填写课程名称、课时、字数限制、附加要求等字段
- 用户仍在原有页面查看生成进度
- 用户仍通过原有位置查看 markdown、预览图、下载按钮和最终生成结果

### 4.2 请求语义保持稳定

新后端必须接受当前前端表达的核心业务语义，包括但不限于：

- 课程名称
- 章节 / 单元 / 课时
- 字数或页数限制
- 附加要求
- 题型、难度、知识点等参数
- 教案、PPT、试卷的组合生成意图

字段名字未来可以在兼容层内部映射，但对前端来说，语义必须保持不变。

### 4.3 响应语义保持稳定

前端目前依赖的结果呈现逻辑必须继续成立，包括但不限于：

- 状态展示
- markdown 正文展示
- PPT 预览图展示
- 下载地址或展示地址
- 流式阶段更新

允许内部结构变化，不允许把前端已依赖的结果能力去掉。

### 4.4 SSE 事件语义保持稳定

当前前端的通用 SSE 客户端在 `/Users/sss/directionai/DirectionAICloud/nginx/src/util/sse-client.ts` 中定义了以下事件类型：

- `thinking_start`
- `thinking_chunk`
- `thinking_end`
- `progress`
- `preview`
- `done`
- `error`

因此，新后端在兼容层输出流式结果时，必须继续支持这些事件语义。

#### Lesson Plan SSE 最低兼容要求

- `done` 阶段必须能提供 `lesson_plan`
- `progress` 需要能够驱动前端思考步骤展示

#### Exam SSE 最低兼容要求

- `done` 阶段至少要能让前端解析出题目数组
- 前端当前兼容的字段包括：
  - `question`
  - `questions`
  - `output.question`

#### PPT SSE 最低兼容要求

- `preview` 阶段应继续支持：
  - `markdown_content`
  - `completed_slides`
  - `total_slides`
  - `current_title`
- `done` 阶段应继续支持：
  - `markdown_content`
  - `display_url`
  - `download_url`
  - `preview_images`
  - `preview_warning`
  - `biz_id`

## 5. 核心用户场景

### 场景一：只生成教案

用户填写课程信息和附加要求，希望得到一份可直接查看或导出的教案。

系统应：

- 理解课程目标和授课对象
- 结合教材、知识库或用户上传资料
- 输出结构完整、逻辑清晰、适配教学场景的教案

### 场景二：只生成试卷

用户填写课程、知识点、题型分布、难度分布等信息，希望直接生成一份结构化试卷。

系统应：

- 先理解试卷蓝图
- 再按题型和难度要求生成题目
- 输出题目、答案、解析、难度、知识点映射

### 场景三：只生成 PPT

用户填写课程信息、页数限制、附加要求，希望得到可下载、可预览的 PPT。

系统应：

- 输出最终 PPT 文件
- 输出前端可展示的 markdown 内容
- 输出预览图和下载地址
- 保留 trace、QA、repair 能力

### 场景五：文档上传后再生成

当前前端和旧系统已经存在“先上传文档，再生成”的路径。

因此新系统应支持：

1. 上传教材、讲义或原始文档
2. 提取文本和摘要
3. 将处理后的结构化上下文用于教案、PPT 或试卷生成

这个能力既可以继续走直接的上传接口，也可以在未来被整合进统一任务入口，但前端层面的上传体验不能消失。

### 场景四：组合任务

用户可能一次性请求多个产物：

- 教案 + 试卷
- 教案 + PPT
- 教案 + PPT + 试卷

系统应能识别任务组合，而不是默认总是把三类产物都生成一遍。

## 6. 三类核心能力要求

### 6.1 教案生成

#### 输入

- 课程名称
- 章节 / 单元 / 课时
- 目标人群
- 字数限制
- 附加要求
- 可选教材、讲义或参考文档
- 可选知识库检索结果

#### 输出

至少应提供：

- 完整的教案正文
- 结构化教案 artifact
- 可供后续 PPT 和试卷复用的中间上下文

推荐最小结构化产物：

- `lesson_plan_markdown`
- `lesson_plan_outline`
- `lesson_plan_metadata`
- `lesson_plan_context_for_downstream`

#### 内容要求

教案应尽可能包含以下关键部分：

- 教学目标
- 重点与难点
- 教学方法
- 教学流程
- 理论内容
- 实践或互动环节
- 总结与作业

#### 工程要求

- 输出结构应稳定，不能每次字段乱变
- 应尽量复用知识库、教材或上传资料
- 应支持后续评估和回炉
- 不要求死守当前 `evoagentx` 的 workflow 结构，可以按 DeerFlow 风格重构
- `done` 事件对前端至少要稳定提供 `lesson_plan`

### 6.2 试卷生成

#### 输入

- 学科 / 课程
- 知识点
- 题型分布
- 难度分布
- 题目数量
- 附加要求
- 可选上游教案或知识上下文

#### 输出

至少应提供：

- 结构化试卷 artifact
- 题目列表
- 每题答案
- 每题解析
- 每题难度标签
- 每题知识点归属

推荐最小结构化产物：

- `exam_blueprint`
- `questions`
- `answers`
- `analyses`
- `knowledge_point_mapping`

#### 内容要求

- 题型数量必须符合配置
- 难度分布尽量符合配置
- 题目不应明显重复
- 解析应具备基本可读性
- 输出应适合后续校验、去重和回归测试

#### 工程要求

- 可以摆脱当前 `evoagentx/exam_generator.py` 的固定流程
- 允许按题型 fan-out 并行生成
- 最终结构必须稳定，不能只返回一段自由文本
- `done` 事件对前端至少要能被解析为题目数组

### 6.3 PPT 生成

#### 输入

- 课程名称或 PPT 主题
- 知识点
- 页数限制
- 附加要求
- 可选教案内容
- 可选教材或原始文档
- 可选 research / image / asset 相关开关

#### 输出

至少应提供：

- 最终 PPT 文件
- markdown 内容
- 预览图
- 下载地址
- 运行 trace

推荐最小结构化产物：

- `ppt_generation_request`
- `ppt_generation_result`
- `ppt_preview_images`
- `ppt_trace_reference`

#### 质量要求

- 视觉效果不得低于当前 `/Users/sss/directionai/directionai_pptagent`
- 必须保留排版质量和可控性
- 必须保留 QA、repair、trace 能力
- 不允许仅因为“框架统一”就把成熟的 PPT 专用逻辑弱化掉

#### 重构原则

- PPT 允许接入 DeerFlow 编排层
- 但不应在 DeerFlow 中被重新做成自由发挥的通用 Agent
- 更推荐作为专用子系统接入
- 对前端 `preview` 和 `done` 字段语义必须稳定

## 7. 组合任务要求

### 7.1 任务组合必须显式识别

系统必须识别用户到底要什么：

- 只要教案
- 只要试卷
- 只要 PPT
- 要两个产物
- 要三个产物

不能因为内部方便，就默认所有请求都走全链路。

### 7.2 上下游关系

默认推荐的数据依赖关系：

- 教案可作为 PPT 的上游语义输入
- 教案可作为试卷蓝图和知识点约束的上游输入
- 试卷通常不反过来成为教案的必需前置
- PPT 通常不是教案或试卷的必需前置

### 7.3 失败处理

组合任务下，一个子任务失败时：

- 应尽量明确告知失败的是哪一部分
- 不应把所有产物都默默吞掉
- 应允许局部失败、局部重试、局部回炉

### 7.4 组合任务的推荐依赖顺序

默认推荐顺序如下：

1. 如果请求里包含教案，先得到 `lesson_plan_outline` 或 `lesson_plan_context`
2. 试卷在可能时复用教案的知识点与目标约束
3. PPT 在可能时复用教案上下文、标题结构或 lesson context
4. 如果用户只请求单一产物，不强制走上游链路

## 8. 非功能性要求

### 8.1 稳定性

- 前端可继续使用
- 长任务可追踪
- 核心状态稳定
- 工具输出结构稳定

### 8.2 可维护性

- 新人只读少量文档就能理解项目
- 新功能和 bug 修复有清晰落点
- schema、状态机、artifact 有统一约束

### 8.3 可调试性

- 能看到关键阶段
- 能知道使用了哪些工具
- 能知道哪一步失败
- 对 PPT 应能保留更细的 trace 和 repair 信息

### 8.5 向后兼容性

重构期间默认采用“双层兼容”思路：

- 前端看到的接口与结果保持稳定
- 新后端内部可以彻底改写

只要前端不需要改页面和消费逻辑，就认为兼容性目标达成。

### 8.4 可扩展性

- 未来可以继续扩展更多教育产物
- 不被当前手写 workflow 和正则解析强绑定

## 9. 非目标

本次重构默认不把以下内容作为首要目标：

- 前端页面重做
- Java 主业务系统整体重构
- 用户服务、监控、Remotion、Nginx 等非 AI 模块重构
- 用统一 Agent 替代所有确定性逻辑

尤其是 PPT，不以“统一成纯 Agent 逻辑”为目标，而以“效果更稳、系统更可维护”为目标。

## 10. 阶段性验收标准

### 第一阶段：设计完成

满足以下条件可视为设计完成：

- 团队对系统目标和边界达成一致
- 能明确回答前端哪些行为被冻结
- 能明确回答教案、试卷、PPT 各自怎么重构
- 能明确回答什么时候先改文档，什么时候先改代码

### 第二阶段：重构完成

满足以下条件可视为重构完成：

- 前端无需改页面即可接入新后端
- 教案和试卷不再依赖旧硬编码 workflow 才能工作
- PPT 迁移后效果不低于 `directionai_pptagent`
- 组合任务可以稳定运行
- trace、错误处理、回归测试具备可维护性
- 当前前端的 lesson / exam / ppt store 无需改核心消费逻辑即可工作

## 11. 需求变更规则

以后如果出现以下变化，必须先修改本文件：

- 用户可见行为变化
- 前端结果语义变化
- 教案、试卷、PPT 的能力定义变化
- 组合任务行为变化
- 验收标准变化

如果只是实现方式改变，但不影响以上内容，则不必先改本文件。
