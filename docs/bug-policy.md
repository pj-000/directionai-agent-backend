# Bug 处理策略

## 1. 文档定位

这份文档的目标，是让团队在遇到 bug 时先判断问题类型，再决定：

- 先改 `requirements.md`
- 先改 `architecture.md`
- 先补测试
- 还是直接改代码

这份文档不是 bug 列表，而是 bug 处理规则。

如果以后团队成员看到问题后总是直接改代码，导致：

- 需求没澄清
- 边界没修正
- 文档和代码长期脱节

就说明这份文档没有被正确执行。

## 2. 总原则

遇到 bug 后，不要立刻下手改代码。

第一步永远是先分类。  
分类错了，修复顺序通常也会错，最后会出现下面这些后果：

- 代码修了，但产品语义还是不清楚
- 代码修了，但模块边界继续恶化
- 代码修了，但没有测试，过一阵又坏
- 明明只是小 bug，却被误升成大范围重构

推荐统一处理顺序：

1. 先确认现象
2. 再判断属于哪一类问题
3. 再决定先改文档、先补测试，还是直接改代码
4. 最后补验证和回归

## 3. 四类问题

## 3.1 第一类：需求 / 设计层问题

这类问题表示：

系统“应该怎么表现”这件事本身不清楚，或者当前定义就是错的。

### 典型信号

- 团队成员对正确结果理解不一致
- 用户和开发对输出的预期不同
- 你自己说不清正确行为是什么
- 修复会改变用户可见行为
- 某个字段、状态、阶段、按钮语义不明确
- 当前实现技术上可行，但产品上不对

### 教育系统里的典型例子

- 用户只请求教案时，系统要不要顺便生成 PPT
- PPT 生成失败时，页面上应显示“部分完成”还是“整体失败”
- 试卷生成失败后，是直接中断还是先返回已成功题型
- 流式阶段里 `completed` 到底表示“AI 已产出结果”，还是“前端也已经渲染完成”
- 用户提供了教材但没有知识点时，系统是否应自动补足知识点
- 旧前端的 `/pptagentapi/stream_ppt` 是否允许改为新的单一任务入口

### 正确处理顺序

1. 先修改 `requirements.md`
2. 如果行为变化会牵涉边界、产物或状态机，再同步更新 `architecture.md`
3. 再修改代码
4. 补测试并联调

### 一句话判断

只要“正确行为要发生变化”，就先改需求文档，再改代码。

## 3.2 第二类：架构 / 契约层问题

这类问题表示：

系统各部分怎么协作，这件事出问题了。

不是产品需求错了，而是边界、依赖方向、schema、状态机、职责划分不稳。

### 典型信号

- 改一个地方总会连带很多地方
- 同一个结果结构在不同地方长得不一样
- 某层做了不该由它做的事
- orchestrator 到处写兼容补丁
- SSE 事件结构经常漂移
- skill 输出格式不稳定
- state 从一个非法状态跳到另一个状态
- 一个问题修完后，另一个模块总被顺手打断

### 教育系统里的典型例子

- Compatibility API 里直接写了教案生成逻辑
- Lead Agent 直接绕过 PPT 子系统输出最终 PPT
- Tool 有时返回 `success/data`，有时返回 `ok/result`
- 教案 artifact 有时是结构化 JSON，有时只是一段 markdown
- `completed -> running` 这类非法状态回跳存在
- 试卷生成器直接依赖了前端表单对象，而不是内部 `TaskSpec`
- 直接在 `backend/packages/harness/deerflow/` 里大段魔改业务代码

### 正确处理顺序

1. 先修改 `architecture.md`
2. 如涉及工程规则或测试策略，也同步修改 `engineering.md`
3. 先补 contract test / integration test
4. 再修改代码
5. 最后验证兼容性

### 一句话判断

只要问题暴露的是边界、依赖、schema、状态机、契约，就先改架构文档，再改代码。

## 3.3 第三类：纯实现问题

这类问题表示：

规范本身没错，只是代码没有做到。

### 典型信号

- 正确行为没有争议
- 看文档就知道实现不符合要求
- 不涉及前端行为变化
- 不涉及模块边界变化
- 不涉及状态机规则变化
- 只是判断条件、路径、await、异常处理、重试、空值等实现写错

### 教育系统里的典型例子

- 某个字段解析漏了空值分支
- 教案导出时文件路径拼错
- 试卷某题型数量统计少 1
- SSE 某阶段少发了一条事件
- PPT 某工具超时后没有正确走 fallback
- 某个异步调用漏了 `await`
- `done` 事件已经定义好，但兼容层忘了把 `lesson_plan` 或 `preview_images` 透给前端

### 正确处理顺序

1. 先复现问题
2. 先补失败测试或回归测试
3. 再修改代码
4. 跑测试
5. 只有当修复过程中发现上层规范不清时，才回头补文档

### 一句话判断

只要规范没变，就先补测试，再修代码。

## 3.4 第四类：纯重构 / 非行为变更

这类不是 bug 修复，而是整理代码。

### 典型信号

- 提取重复逻辑
- 重命名变量或函数
- 拆分文件
- 优化目录结构
- 提升可读性
- 不改外部行为

### 教育系统里的典型例子

- 把多个 lesson service 拆成更清晰的文件
- 合并重复的 tool error 处理函数
- 把 prompt loader 从 orchestrator 提到独立模块
- 重命名 `helper.py` 为更清晰的职责文件

### 正确处理顺序

1. 直接改代码
2. 跑现有测试
3. 如果发现行为变化，立刻停止，并重新按前三类问题分类

### 一句话判断

不改行为的重构，可以直接改代码，但必须靠测试兜底。

## 4. 快速决策树

当你看到一个 bug，可以按下面顺序判断：

### 第一步：正确行为是否清楚

如果不清楚：

- 先看 `requirements.md`
- 先改需求定义

### 第二步：模块边界或契约是否暴露问题

如果是：

- 先看 `architecture.md`
- 先修边界和契约定义

### 第三步：规范已清楚，只是实现没做好

如果是：

- 先看 `engineering.md`
- 先补测试，再改代码

### 第四步：只是整理代码

如果是：

- 直接改代码
- 用现有测试兜底

## 5. 快速决策表

### 先改 `requirements.md`

适用于：

- 用户看到的结果要变
- 产品语义要变
- 业务流程定义要变
- 某个阶段、按钮、状态的业务含义要变

### 先改 `architecture.md`

适用于：

- 模块边界要变
- 依赖方向要变
- 状态机规则要变
- API、artifact、SSE、tool、skill 契约要变
- DeerFlow 上游区与 DirectionAI 自有代码区的边界要变

### 先补测试

适用于：

- 条件判断错误
- parser / validator 错误
- 空值 / 边界值错误
- await / timeout / retry 错误
- 已明确为纯实现 bug

### 直接改代码

适用于：

- 纯重构
- 外部行为不变
- 现有测试足以兜底

## 6. 复杂 bug 记录规则

不是所有 bug 都值得写专门文档。

只有下面这类 bug，才建议单独沉淀成 bug 记录：

- 会反复出现
- 根因不直观
- 跨多个模块联动
- 影响架构或契约
- 修复方式需要长期记住

不适合单独记文档的情况：

- 拼写错误
- 一次性小修补
- 明显的判断写反
- 很快修好的低风险实现错误

## 7. 复杂 bug 模板

如果以后需要单独记录复杂 bug，推荐用下面这个模板。

### 标题

一句话描述问题。

### 现象

系统或用户看到的异常表现。

### 影响范围

- 影响哪些流程
- 是否影响前端展示
- 是否影响历史数据或已有产物

### 复现条件

稳定复现所需的输入、环境和步骤。

### 根因

根因必须写清楚是：

- 需求定义问题
- 架构边界问题
- 纯实现问题
- 还是多因素共同触发

### 修复原则

修复时必须遵守哪些边界和约束。

### 回归测试入口

写明新增或修改的测试位置。

### 是否需要先改文档

- 是否影响 `requirements.md`
- 是否影响 `architecture.md`
- 是否影响 `engineering.md`

## 8. 团队执行要求

以后看到 bug，不允许直接跳过分类步骤。

至少要先在心里回答这 3 个问题：

1. 正确行为清楚吗
2. 是不是边界、状态、schema 的问题
3. 这次修复是否必须先补测试

如果这 3 个问题都没回答清楚，就不应该直接改代码。

## 9. 当前仓库的补充判断规则

### 9.1 改动发生在 `backend/packages/harness/deerflow/`

默认先问自己：

- 这是上游核心问题吗
- 能不能改在 `backend/packages/directionai/` 或 `backend/app/gateway/routers/`

如果能绕开上游核心，就不要直接侵入修改。

### 9.2 改动发生在 Compatibility API

默认先问自己：

- 前端消费字段有没有变化
- SSE 事件名和 payload 有没有变化

如果变化了，通常不是纯实现问题，而是需求或契约问题。

### 9.3 改动发生在 PPT 子系统

默认先问自己：

- 这次修复会不会伤到视觉质量
- 这次修复只是实现 bug，还是在改变 PPT 子系统和 DeerFlow 的边界

如果会改变边界，先改 `architecture.md`。  
如果只是修某个渲染缺陷，先补回归测试再改代码。
